02

GPU Bubble 问题

问题根源：各 Rank 独立随机采样，时长分布不均，Packing 后 packed cuts 数量不一致，总计算量不同导致同步等待。
图 1: GPU Bubble 问题示意图
📊
关键洞察

当不同 GPU 处理的 packed cuts 数量不均衡时，处理数据量少的 GPU 会提前完成，但梯度同步需要等待所有 GPU 完成。这些等待时间就是 GPU Bubble，严重降低了多 GPU 训练的效率。

四种 Packing 模式

none
lhotse_legacy
bestfit
knapsack
None 模式：不启用任何 packing，直接使用原始音频片段进行训练，padding 浪费严重。
图 3: None 模式 - Padding 浪费示意
参数 默认值 说明
--packing-method none 不启用 packing
--concatenate-cuts False 不拼接 cuts
Lhotse Legacy 模式：每个 Rank 独立进行 CutConcatenate，容易产生 DDP Bubble。

各 Rank 独立采样和 packing，导致不同 GPU 的 packed cuts 数量和时长不一致，产生负载不均。

图 3b: Lhotse Legacy 模式 - DDP Bubble 问题
BestFit 模式：DDP Pack Sampler + Best-fit 装箱算法，确保各 Rank 负载均衡。

使用 Best-fit 算法进行装箱，每次选择能 fit 进剩余空间的最长音频，最大化填充率。

图 3c: BestFit 装箱算法
Knapsack 模式（推荐）：在 Best-fit 基础上，对尾部剩余空间使用 Knapsack 动态规划算法进一步优化。

当剩余时长 ≤ 5s 时，使用 0-1 Knapsack 选择最优组合填充，最大化利用率。

图 3d: Knapsack 动态规划优化尾部填充

Block-Diagonal Attention Mask

重要：Block-diagonal mask 只能阻止 Attention 层面的信息泄漏，不能阻止 CNN 的信息泄漏！对于深层 CNN，1s gap 可能不够。

关键参数配置

参数 默认值 说明
--packing-method knapsack Packing 模式: none, lhotse_legacy, bestfit, knapsack
--max-duration 1500 每 rank raw cuts 总时长上限 (秒)
--max-cuts 50 每 rank packed cuts 数量 (固定 shape)
--concatenate-cuts-max-duration 30 单条 packed cut 时长上限 (秒)
--gap 1.0 utterance 间隔 (秒)
--causal 1 启用流式/因果模式
--chunk-size "16,32,64,-1" Chunk 大小候选 (帧, @50Hz)
--left-context-frames "64,128,256,-1" 左上下文帧数候选
# 推荐配置 (run_french_150M.sh) # ============ Packing 配置 ============ --packing-method knapsack # 推荐: knapsack 模式 --max-duration 1500 # 每 rank 时长上限 --max-cuts 50 # 固定 packed cuts 数量 --concatenate-cuts-max-duration 30 # 单条 packed cut 上限 --gap 2.0 # 增大 gap 减少 CNN 泄漏 # ============ 流式配置 ============ --causal 1 # 启用流式训练 --chunk-size "16,32,64,-1" # chunk 候选 --left-context-frames "64,128,256,-1" # 左上下文候选 # ============ 数据加载配置 ============ --num-workers 16 # DataLoader workers --prefetch-factor 16 # 预取因子
