02

GPU Bubble 问题

问题根源：各 Rank 独立随机采样，时长分布不均，Packing 后 packed cuts 数量不一致，总计算量不同导致同步等待。
图 1: GPU Bubble 问题示意图
📊
关键洞察

当不同 GPU 处理的 packed cuts 数量不均衡时，处理数据量少的 GPU 会提前完成，但梯度同步需要等待所有 GPU 完成。这些等待时间就是 GPU Bubble，严重降低了多 GPU 训练的效率。

None 模式：不启用任何 packing，直接使用原始音频片段进行训练，padding 浪费严重。
图 3: None 模式 - Padding 浪费示意
参数 默认值 说明
--packing-method none 不启用 packing
--concatenate-cuts False 不拼接 cuts
Lhotse Legacy 模式：每个 Rank 独立进行 CutConcatenate，容易产生 DDP Bubble。

各 Rank 独立采样和 packing，导致不同 GPU 的 packed cuts 数量和时长不一致，产生负载不均。

图 3b: Lhotse Legacy 模式 - DDP Bubble 问题
BestFit 模式：DDP Pack Sampler + Best-fit 装箱算法，确保各 Rank 负载均衡。

使用 Best-fit 算法进行装箱，每次选择能 fit 进剩余空间的最长音频，最大化填充率。

图 3c: BestFit 装箱算法
Knapsack 模式（推荐）：在 Best-fit 基础上，对尾部剩余空间使用 Knapsack 动态规划算法进一步优化。

当剩余时长 ≤ 5s 时，使用 0-1 Knapsack 选择最优组合填充，最大化利用率。

# zipformer.py: _get_attn_mask - 流式训练的 mask 生成
def _get_attn_mask(self, x, chunk_size, left_context_chunks):
    # 计算 chunk id: c = t // chunk_size
    c = t // chunk_size

    # Mask 未来 + 超过限制的过去
    attn_mask = torch.logical_or(
        src_c > tgt_c,                           # 未来 chunk
        src_c < tgt_c - left_context_chunks      # 过久远的历史
    )
    return attn_mask

Node Producer 与 Batch Replay

架构升级：从各 Rank 独立采样过渡到集中式 Node Producer 架构，通过 IPC 队列实现跨 Rank 批次共享，配合 Batch Replay 机制解决 GPU Bubble 问题。

核心组件

🔄
NodeBatchProducer

独立进程/线程中运行，负责统一采样和 Packing

  • 使用 PackAwareSampler 采集 raw cuts
  • CutConcatenate 拼接成 packed cuts
  • 均衡分配算法确保各 Rank 时长一致
  • 将 batch 放入 IPC 队列
📥
NodeBatchIPC

进程间通信的共享队列和同步机制

  • rank_queues: 每个 Rank 的专属队列
  • shared_metrics: 跨进程指标共享
  • stop_event: 停止信号同步
  • lock: 队列操作锁
👤
ConsumerCutSampler

各 Rank 的消费者，从队列获取数据

  • 每个 Rank 一个实例
  • 从对应 queue 获取 batch
  • 支持 Batch Replay 机制
  • 实现 PyTorch Sampler 接口

Pack-Aware 分割流程

图 7: 三步 Pack-Aware 批次生成与分配
均衡策略：使用贪心算法，优先保证各 Rank 的 packed cuts 数量相同，然后按时长调整使得总时长尽可能均衡。

Batch Replay 机制

核心概念：Batch Replay 是 ConsumerCutSampler 的三级容错机制中的第二层，当从队列获取 batch 超时（默认 500ms）时触发，通过复用历史 batch 避免训练中断。
图 8: ConsumerCutSampler 三级容错机制
生产环境配置：正式训练时建议使用阻塞模式（block_on_empty=True, block_timeout_sec=60），确保数据流的确定性和可复现性。Batch Replay 机制（replay_on_empty）主要用于开发测试和验证阶段，或作为防止训练中断的保险措施。

📊
关键观察指标

队列长度差异

各 Rank 队列长度应保持在 max_pending_batches 附近。如果某个 Rank 队列持续为空，说明该 Rank 消费过快。

Replay 频率

replay_count / batches_consumed 应该 < 1%。如果过高，说明 Producer 速度不足或配置不均衡。

Producer 生存状态

is_producer_alive 应始终为 true。如果变为 false，说明 Producer 进程崩溃，需要重启训练。

Batch 时间戳

last_batch_time 应持续更新。如果停止更新，说明 Producer 可能陷入死锁或数据耗尽。
